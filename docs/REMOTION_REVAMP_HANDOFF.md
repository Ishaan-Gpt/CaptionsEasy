# Remotion Revamp — Handoff / Continuation Plan

**Read this first in a fresh session.** This document is self-contained — it assumes zero memory of prior conversation. It captures what's done, what's left, architectural decisions already made (and why), and hard-won gotchas so they aren't rediscovered the expensive way.

## 1. Project shape (context a fresh session needs)

- Monorepo: `apps/backend` (FastAPI + Celery), `apps/frontend` (Next.js), `apps/remotion-pipeline` (Remotion render project), `packages/contracts/python` (shared Pydantic schemas).
- Two rendering engines exist side by side:
  - **ASS/libass** (`apps/backend/app/render/engine.py`, `generate_ass()`) — burns subtitles into video via FFmpeg. Original/legacy path.
  - **Remotion** (`apps/remotion-pipeline/src/Subtitles.tsx`, invoked via `engine.py`'s `render_remotion()`) — renders a transparent overlay via headless Chromium, composited onto the source video with FFmpeg.
- `RenderEngine.render()` (`apps/backend/app/render/engine.py`) routes: `is_advanced_template` (currently `{cinematic_emerald, staggered_3line, glow_stack, cartoon_stack, serif_pop}`) → Remotion; everything else (`word_by_word`, `sentence_highlight`, `sentence_clean`) → ASS.
- **5 "advanced" templates were fully rebuilt in Remotion this session**, replacing what used to be ASS-only, hand-tuned-per-template renderers. Each has a dedicated branch in `Subtitles.tsx` (`if (template === "...")  { ... }`).
- **Local dev servers**: backend `uvicorn app.main:app --reload` on port 8000 (venv at `apps/backend/.venv`, must set `PYTHONPATH` to repo root or `packages.contracts.python` import fails — see gotcha below), frontend `pnpm --filter frontend dev` on port 3000. Frontend's `.env.local` `NEXT_PUBLIC_API_URL` MUST point at `:8000` — it was previously misconfigured to `:8001` (a stale process) and cost a long debugging session; verify this first if "nothing seems to work."

### 2b. Critical bugs found and fixed (the SAME bug pattern recurred 3 times — know this pattern)
**The pattern**: some per-word/per-frame value (hero index, font size, an animation timer) was derived from "whichever word/highlight is currently active" instead of "this specific word/card's own fixed identity." Every time ANY word's highlight changed, the wrongly-scoped value changed too, causing something that should have been stable to visibly re-trigger, reshuffle, or reflow. Occurred 3 times:
1. **`cinematic_emerald` hero picking the wrong split point** — used the *currently active* highlight's index as the hero/split point instead of the *fixed* `is_keyword`-flagged highlight for the whole card. Result: the 3-word layout completely reshuffled itself once per word. Fixed by resolving `heroIndex` once from `cardHighlights.find(h => h.payload.is_keyword)`, never from `highlightedIndex`.
2. **`cinematic_emerald` hero pop-in animation replaying** — the entrance spring was keyed to `elapsedHighlightFrames` (from *whichever* highlight is currently active), so it reset and replayed every time ANY word in the card lit up, not just the hero. Fixed with a separate `heroPopSpring` keyed to `heroEvent.start_ms` specifically.
3. **Generic `word_by_word` branch reflow (found in Phase A verification, this session)** — the active/keyword word's `fontSize` (not just its `transform: scale`) was bumped when it became active. `fontSize` affects real layout, so the whole row reflowed every time the active word changed — and once Phase A added a background pill box, that reflow visibly dragged the pill along with it, reading as "the caption card regenerating multiple times." Fixed by making the keyword's enlarged look purely `transform: scale()` (visual-only, no layout impact) — `wordSize` is now always the base `size`, never multiplied.

**Also fixed**:
- **Progressive reveal must never remove words from the DOM** (array-filter+join). Must use `visibility`/`opacity` toggling so each word's *layout slot* is reserved from frame one and it just becomes visible in place — otherwise the line's rendered width grows as each word is spliced in, which can shift the whole block. This is the `RevealLine` component at the top of `Subtitles.tsx` — reuse it, don't reinvent per-word reveal logic.
- **VP9 alpha silently broken** in this Remotion/Chromium build — `--codec=vp9 --pixel-format=yuva420p` is accepted but produces an opaque video anyway (a real, confirmed bug in this environment, not a misconfiguration). The overlay compositing step in `engine.py`'s `render_remotion()` uses **ProRes 4444** (`--codec=prores --prores-profile=4444 --pixel-format=yuva444p10le`) instead — verified via `ffprobe` to actually carry alpha end-to-end. Do not switch back to VP9 without re-verifying.
- **`probe_metadata()` bug**: requested `stream=...,duration` from ffprobe but read the value back from `format.duration`, which was never requested — always silently returned `0.0`, which made `duration_frames` round to 0/1, which made the Remotion overlay render a single blank frame — captions vanished entirely with an otherwise "successful" render. Fixed by requesting `format=size,duration` and reading the right key. **If a Remotion export produces a real duration video with zero captions, check this first.**
- **`shell=True` + list argv on `subprocess.run`** for the Remotion CLI call only works correctly on Windows (list2cmdline). On POSIX/Linux (production), it silently drops all arguments except `npx`. Fixed with `shell=(os.name == "nt")`.
- **Backend job pipeline could report "completed" with zero actual output**: `_build_stages()` in `apps/backend/app/worker/tasks.py` used to silently fall back to `build_dummy_stages()` (a no-op stub) whenever a job row lookup failed or `job_type` didn't match, and the pipeline runner had no way to tell a no-op stage list from a real one — it called `mark_completed()` regardless. Fixed by raising `UnroutableJobError` instead, routed through proper `mark_failed`/retry handling.
- **Frontend never showed the actual rendered export** — main preview `<video src>` always bound to the raw uploaded clip; the export's URL was dead-code fallback. Fixed: `activeExportId` state, auto-switches to the finished export after a render completes, "Back to Live Preview" toggle, "Preview" button per row in Render History.

### 2c. Font handling
- Fonts must be registered via `FontFace` + `delayRender`/`continueRender` in `apps/remotion-pipeline/src/fonts.ts` — **multi-word family names must be quoted** (`"'Baloo 2', sans-serif"`, not `"Baloo 2"`) or Chromium silently falls back to a system serif font with zero error (cost real debugging time — the symptom was a wrong-looking font in a rendered frame, no error anywhere).
- Font files live in **two places that must stay in sync**: repo-root `fonts/` (used by the ASS/FFmpeg `fontsdir=` path) and `apps/remotion-pipeline/public/fonts/` (served to Remotion via `staticFile()`). Any new font needs copying to both.

### Phase C — Bounding-box visual editor (the big one)
**Architecture already decided** (researched via a dedicated investigation this session — don't re-derive):
- `MotionScript` rows are fully regenerated/re-inserted on every style change (`generate_motion_script` and `export_project` in `apps/backend/app/api/v1/projects.py` both call `DummyRenderPlanProvider.plan()` fresh). `evt-N` timeline-event IDs are a local counter that restarts at 1 every call — **not stable across regenerations**. Any override keyed by `evt-N` would silently detach the moment word grouping changes even slightly.
- The most durable anchor available is **the caption card's own `start_ms`** (its first word's Whisper timestamp — deterministic given the same transcript).
- `CustomStyleRequest`/`presets.json` is a full-replace blob per project, not a partial-merge JSON column — **do not** add a sparse per-fragment map inside it; an unrelated save would silently wipe it.
- `GlobalSettings.safe_area` (`SafeArea`: top/bottom/left/right, `packages/contracts/python/render_plan.py`) is **already the existing global-bounding-box concept** — reuse it for the global default, don't invent a parallel field.
- All contract models are `StrictModel` (`extra="forbid"`) — every new field must be declared explicitly.

**Design**: add `box: SafeArea | None` to `CaptionPayload`. New `Project.fragment_overrides_json` JSONB column (new Alembic migration) stores a sparse map `{ "<start_ms_bucket>": { box: {...} } }`. Immediately after `DummyRenderPlanProvider.plan()` builds the timeline (both call sites), a merge step resolves each caption event's `payload.box` = matching fragment override (nearest `start_ms` within ~±250ms tolerance) **or** the project's global box default **or** `None` (existing 90%/800px fallback behavior). Renderers (both ASS and Remotion) then just read `caption.payload.box` uniformly — no separate override-fetching logic needed anywhere else.

- Backend: migration; `PUT /projects/{id}/fragment-override` (upsert, keyed by `start_ms`) + `DELETE /projects/{id}/fragment-override/{start_ms}`; extend `CustomStyleRequest`/response with global `box_top/bottom/left/right` mapped onto `GlobalSettings.safe_area`; implement the merge step at both `generate_motion_script` and `export_project`.
- Frontend: new draggable/resizable overlay on the video preview (reuse the existing `S` canvas-scale factor already used for the caption overlay in `page.tsx`) — **user explicitly wants width, height, AND position all draggable** (not width-only). Corner/edge handles for width+height, a center-drag handle for position, pixel-delta → canvas-relative-percentage conversion. On drag-end: "Apply to: This Caption / All Captions" — routes to the new fragment-override endpoint or the existing `saveCustomStyle` global path respectively (**user explicitly wants this per-fragment-vs-global choice**, not global-only).
- Text reflow: `word_by_word`/`sentence_highlight`/`sentence_clean` already use `flex-wrap` — feed the box's live width into that wrap container and reflow already works correctly (this is just plumbing, wrapping logic already exists). The 5 advanced (fixed 3-line) templates continue to shrink-to-fit on box-width change rather than introducing new wraps — their per-line layout can't wrap without breaking the template's identity; state this limitation plainly rather than overpromising full reflow there.
- Verify: drag the box narrower on a `word_by_word` caption, confirm text wraps to an extra line in the live preview; wider, confirm it unwraps; set a per-fragment override, scrub to a different caption, confirm it reverts to the global box; re-render through Remotion and confirm the box actually constrained the export, not just the preview.
